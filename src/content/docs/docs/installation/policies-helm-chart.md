The `kyverno/kyverno-policies` Helm chart deploys Kyverno policies (Pod Security Standards and others) into your cluster. In addition to the core options such as `validationFailureAction` and `podSecurityStandard`, the chart exposes several advanced configuration features introduced in chart version 1.18. This page documents those features.

For general installation of the policies chart, see the [Installation page](/docs/installation/installation#pod-security-policies-optional).

## Excludes for ValidatingPolicy

When `policyType` is set to `ValidatingPolicy`, the chart supports configuring resource excludes per policy. These excludes are translated into native `ValidatingAdmissionPolicy` exclude rules and CEL `matchConditions` on the generated `ValidatingPolicy` resources.

:::note[Note]
The `vpolExclude` and `vpolExcludeByPolicy` values only apply when `policyType: ValidatingPolicy`. For `ClusterPolicy`, use `policyExclude` instead.
:::

### Global excludes (`vpolExclude`)

Use `vpolExclude` to define excludes that apply to **all** `ValidatingPolicy` resources generated by the chart. The following sub-keys are supported:

| Key                    | Type                                              | Description                                                                                              |
| ---------------------- | ------------------------------------------------- | -------------------------------------------------------------------------------------------------------- |
| `excludeNamespaces`    | `[]string`                                        | Namespace names to exclude. Generates a CEL `matchCondition` that skips the policy for these namespaces. |
| `excludeSubjects`      | `[]rbac.Subject`                                  | Users, groups, or ServiceAccounts to exclude. Generates a CEL `matchCondition`.                          |
| `excludeResourceRules` | `[]admissionregistration.NamedRuleWithOperations` | Native VAP `excludeResourceRules` entries passed through directly.                                       |
| `matchConditions`      | `[]admissionregistration.MatchCondition`          | Custom CEL `matchConditions` passed through directly for advanced filtering.                             |

Example — exclude the `kube-system` namespace and all requests from `system:masters` from every policy:

```yaml
vpolExclude:
  excludeNamespaces:
    - kube-system
  excludeSubjects:
    - kind: Group
      name: system:masters
```

### Per-policy excludes (`vpolExcludeByPolicy`)

Use `vpolExcludeByPolicy` to define excludes for a **specific** policy. When a policy is listed here, its entry **completely replaces** the global `vpolExclude` defaults for that policy; the two are not merged.

Each key under `vpolExcludeByPolicy` is a policy name, and its value supports the same sub-keys as `vpolExclude`.

Example — exclude `kube-system` and `monitoring` namespaces only for `disallow-host-path`, while keeping global excludes for all other policies:

```yaml
vpolExclude:
  excludeNamespaces:
    - kube-system

vpolExcludeByPolicy:
  disallow-host-path:
    excludeNamespaces:
      - kube-system
      - monitoring
    excludeSubjects:
      - kind: Group
        name: system:masters
      - kind: ServiceAccount
        namespace: kube-system
        name: default
    excludeResourceRules:
      - apiGroups:
          - ''
        apiVersions:
          - v1
        operations:
          - CREATE
          - UPDATE
        resources:
          - pods
        resourceNames:
          - specific-pod-name
    matchConditions:
      - name: exclude-temp-pods
        expression: "!object.metadata.name.startsWith('temp-')"
```

### How excludes compose

- `vpolExclude` defines a baseline set of excludes shared by all policies.
- `vpolExcludeByPolicy.<policyName>` defines the **full** exclude configuration for a single policy, overriding — not extending — the global defaults for that policy.
- Policies that are not listed in `vpolExcludeByPolicy` use the global `vpolExclude` configuration.

## Audit Annotations for ValidatingPolicy

[Audit annotations](https://kubernetes.io/docs/reference/access-authn-authz/validating-admission-policy/#audit-annotations) allow a `ValidatingAdmissionPolicy` to record additional key-value data into the Kubernetes API server audit log when a request is evaluated. The `kyverno-policies` chart exposes this feature through two values when `policyType: ValidatingPolicy`.

:::note[Note]
`auditAnnotations` and `auditAnnotationsByPolicy` only apply when `policyType: ValidatingPolicy`.
:::

### Global audit annotations (`auditAnnotations`)

Use `auditAnnotations` to set audit annotations that are added to **all** generated `ValidatingPolicy` resources. The value is a map where each key is the annotation name and each value is a CEL expression that is evaluated at admission time to produce the annotation value.

```yaml
auditAnnotations:
  policy-evaluated-by: "'kyverno-policies-helm'"
```

The CEL expression must return a `string`. The annotation appears in the API server audit log entries for requests that are evaluated by the policy.

### Per-policy audit annotations (`auditAnnotationsByPolicy`)

Use `auditAnnotationsByPolicy` to set audit annotations on a **specific** policy. Entries here are merged with the global `auditAnnotations`; if the same key appears in both, the per-policy value takes precedence.

```yaml
auditAnnotationsByPolicy:
  disallow-host-path:
    violation-details: >-
      has(object.spec.volumes)
        ? 'hostPath volumes found: ' + object.spec.volumes.filter(v, has(v.hostPath)).map(v, v.name).join(', ')
        : 'no volumes'
```

With the configuration above, denied admission requests for `disallow-host-path` will include a `violation-details` annotation in the audit log describing exactly which volumes triggered the denial.

## Per-Policy Custom Annotation Overrides

The chart allows setting arbitrary custom annotations on all generated policy resources via `customAnnotations`. In chart version 1.18, support was added for overriding or extending those annotations on a **per-policy** basis through `customAnnotationsByPolicy`.

### Global custom annotations (`customAnnotations`)

```yaml
customAnnotations:
  team: platform
  managed-by: kyverno-helm
```

### Per-policy annotation overrides (`customAnnotationsByPolicy`)

Each key under `customAnnotationsByPolicy` is a policy name. Its value is a map of annotation key/value pairs. Per-policy entries are merged with the global `customAnnotations`; a per-policy key overrides the same global key for that policy only.

```yaml
customAnnotations:
  team: platform

customAnnotationsByPolicy:
  disallow-host-path:
    team: security
    severity-override: critical
```

In the example above, all policies except `disallow-host-path` receive `team: platform`. The `disallow-host-path` policy receives `team: security` and the additional `severity-override: critical` annotation.

## Full Example

The following `values.yaml` snippet combines all three features for a `ValidatingPolicy`-based deployment:

```yaml
policyType: ValidatingPolicy

# Global excludes for all ValidatingPolicy resources
vpolExclude:
  excludeNamespaces:
    - kube-system
    - kyverno

# Per-policy excludes (replaces global vpolExclude for this policy)
vpolExcludeByPolicy:
  disallow-host-path:
    excludeNamespaces:
      - kube-system
      - monitoring
    excludeSubjects:
      - kind: Group
        name: system:masters

# Global audit annotation recorded in the API server audit log
auditAnnotations:
  policy-evaluated-by: "'kyverno-policies-helm'"

# Per-policy audit annotation with a detailed CEL expression
auditAnnotationsByPolicy:
  disallow-host-path:
    violation-details: >-
      has(object.spec.volumes)
        ? 'hostPath volumes found: ' + object.spec.volumes.filter(v, has(v.hostPath)).map(v, v.name).join(', ')
        : 'no volumes'

# Global custom annotation on all policies
customAnnotations:
  team: platform

# Override custom annotations for a single policy
customAnnotationsByPolicy:
  disallow-host-path:
    team: security
    severity-override: critical
```
